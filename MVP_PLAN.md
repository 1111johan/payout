# Amazon 提现自动化 MVP 方案

版本：v0.1
日期：2026-07-13

## 1. MVP 的目标

做一个仅供店铺所有者本人使用的本地管理工具，让用户无需编写命令即可：

1. 检查 Amazon SP-API 凭证是否有效。
2. 查看官方支持站点的收款方式。
3. 在预演模式下测试提现流程。
4. 为受支持的欧洲站设置每日检查和自动提现。
5. 查看每次执行的时间、站点、结果和错误原因。
6. 对英国、美国等不支持主动提现的站点，仅提供余额监控和人工操作提醒。

MVP 默认在 Mac 本机运行，默认只监听 `127.0.0.1`，不暴露到互联网。真实提现默认关闭。

## 2. 官方能力边界

Amazon Transfers API 当前支持：

- 比利时（BE）
- 德国（DE）
- 西班牙（ES）
- 法国（FR）
- 意大利（IT）
- 荷兰（NL）
- 波兰（PL）
- 瑞典（SE）

英国（UK/GB）、美国（US）、加拿大（CA）不能通过 Transfers API 发起提现。官方文档：

https://developer-docs.amazon.com/sp-api/docs/transfers-api

另外，Transfers API 不允许输入一个自定义提现金额。程序只能请求 Amazon 将当时符合条件的余额付到 Seller Central 中已经设置好的默认收款方式。

因此 MVP 分为两种工作模式：

| 站点类型 | MVP 行为 |
| --- | --- |
| BE、DE、ES、FR、IT、NL、PL、SE | 查询收款方式、预演、在批准后自动发起提现 |
| UK、US、CA 等 | 监控余额并提醒用户前往 Seller Central 手动操作，不模拟网页登录点击 |

## 3. 最小功能范围

### 3.1 必须包含

- 单个 Amazon 卖家账户。
- 一个本地网页控制台。
- 凭证状态检查，但页面永远不回显 Client Secret 和 Refresh Token。
- 显示当前运行模式：沙盒/生产、预演/真实。
- 显示支持的八个欧洲站点。
- 查询站点收款方式。
- 手动执行一次预演。
- 设置每日执行时间和站点。
- 每个站点保存最近一次执行结果。
- SQLite 本地运行记录。
- API Key 本地访问保护。
- 幂等键防止重复提交。
- 每个站点至少间隔 24 小时再次提交。
- 真实提现需要四重开关、站点确认和生产凭证。

### 3.2 MVP 暂不包含

- 多卖家、多用户和权限管理。
- 手机 App。
- Amazon Appstore 公开上架。
- 让其他卖家在线授权。
- 指定提现金额。
- 自动修改银行账户。
- 英国或美国站的自动网页点击、Cookie 登录、验证码或 MFA 绕过。
- 将接口直接开放到公网。
- 财务对账、税务或会计报表。

## 4. 用户操作流程

```text
打开本地控制台
    ↓
查看服务与凭证状态
    ↓
选择一个支持的欧洲站点
    ↓
查询默认收款方式
    ↓
执行预演（不会提现）
    ↓
完成生产应用、角色和卖家授权
    ↓
人工真实测试一次
    ↓
开启单站点每日自动任务
    ↓
查看执行历史和失败原因
```

## 5. 页面设计

MVP 只需要一个本地网页，分成四块：

### 状态区

- 服务是否运行。
- `sandbox` 或 `production`。
- `DRY_RUN` 是否开启。
- Amazon 凭证是否完整。
- 最近一次 Amazon 连接测试结果。

### 站点区

- 八个支持站点列表。
- 当前选择站点。
- 收款方式查询按钮。
- 明确显示“不支持英国/美国主动提现”。

### 自动任务区

- 是否启用每日任务。
- 每日执行时间。
- 要执行的站点。
- 保存按钮。
- 下次预计执行时间。
- 生产模式下显示醒目的风险确认。

### 历史记录区

- 执行时间。
- 站点。
- 触发来源：手动或定时。
- 状态：预演、已提交、失败、跳过。
- Amazon 请求编号或简化错误信息。
- 不显示密钥、Token、银行账号完整号码。

## 6. 技术结构

```text
Mac 浏览器
    ↓ 仅访问 127.0.0.1
本地网页控制台
    ↓
现有 Python REST API
    ├── Amazon LWA 登录换取短期 Access Token
    ├── Amazon Transfers API
    ├── 每日任务调度器
    └── SQLite 执行历史与防重复记录
```

继续使用 Python 标准库，MVP 不强制安装第三方依赖。Mac 定时运行优先使用 `launchd`，而不是依赖一直打开终端窗口。

## 7. 在现有接口上增加的能力

现有接口继续保留：

- `GET /health`
- `GET /v1/marketplaces`
- `GET /v1/payment-methods`
- `POST /v1/payouts`

MVP 增加：

- `GET /`：本地控制台。
- `GET /v1/status`：返回安全的配置状态，不返回凭证内容。
- `POST /v1/credentials/test`：测试 LWA/SP-API 连接。
- `GET /v1/schedules`：查看自动任务。
- `PUT /v1/schedules/{marketplace}`：保存单站点任务。
- `DELETE /v1/schedules/{marketplace}`：关闭任务。
- `GET /v1/payouts/history`：查看执行历史。

## 8. 自动任务规则

每分钟检查一次是否有到期任务，但一次任务必须满足全部条件：

1. 站点属于官方支持范围。
2. 任务已启用。
3. 距离该站点上一次成功提交已超过 24 小时。
4. 当前拥有有效生产凭证。
5. `SP_API_MODE=production`。
6. `DRY_RUN=false`。
7. `ALLOW_PRODUCTION=true`。
8. `ALLOW_PAYOUT_POST=true`。
9. 该站点在明确的自动提现白名单中。

任一条件不满足时只记录“跳过”，不能尝试绕过。

每次任务生成唯一幂等键，例如：

```text
auto-DE-20260713
```

即使程序重启或重复执行，也不能对同一站点重复提交同一任务。

## 9. 上线安全门槛

真实提现前必须完成：

- 在 Amazon 创建并获批生产应用，沙盒凭证不能用于真实账户。
- 应用具有 `Finance and Accounting` 或 `Payment Initiation Service Provider` 角色。
- 卖家账户完成对生产应用的授权并取得生产 Refresh Token。
- Seller Central 中的默认收款方式已经人工核对。
- 连续完成沙盒测试。
- 连续运行至少 7 天预演任务且没有重复记录。
- 人工执行至少 1 次小范围真实测试。
- 首次自动化只启用 1 个支持站点。
- 保留关闭程序和撤销 Refresh Token 的应急方法。

## 10. 验收标准

满足以下条件才算 MVP 完成：

- 用户能在 Mac 浏览器打开本地控制台。
- 控制台不会显示任何完整密钥。
- 用户能看到沙盒/生产和预演/真实状态。
- 用户能查询德国站等受支持站点的收款方式。
- 预演执行不会调用 Amazon 提现 POST。
- 重复幂等键不会产生第二次提交。
- 英国、美国站会被明确拒绝自动提现。
- 可以创建、查看、关闭每日任务。
- 程序重启后任务和历史仍然存在。
- Mac 睡眠或关机期间不会谎报任务成功。
- 所有自动测试通过。

## 11. 实施顺序

### 阶段 A：本地沙盒 MVP

- 配置 `.env`。
- 验证 `/health`。
- 测试 Amazon 沙盒连接。
- 增加本地控制台。
- 增加任务和历史接口。
- 运行预演调度器。

### 阶段 B：生产准备

- 创建或升级生产应用。
- 申请正确角色。
- 完成卖家授权。
- 只读查询真实收款方式。
- 保持真实提现关闭。

### 阶段 C：受控上线

- 手动真实执行一个站点。
- 核对 Seller Central 和银行入账状态。
- 运行 7 天人工确认模式。
- 只为一个站点开启自动任务。

### 阶段 D：稳定运行

- 使用 Mac `launchd` 开机启动。
- 如果需要 24 小时可靠运行，再迁移到私人服务器。
- 增加失败通知、备份和监控。
